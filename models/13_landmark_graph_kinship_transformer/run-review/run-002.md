## Mechanism diagnosis

**Why AUC didn't reach the 0.870 graduation bar:**

1. **The 14×14 → ROIAlign → AdaptiveAvgPool collapses the spatial gain.** The current `LandmarkROITokenizer` runs ROIAlign at 3×3, then pools to 1×1. With stage 3's 14×14 map (vs stage 4's 7×7), each landmark ROI samples ~4× more grid cells, but the immediate global average pool throws that detail away. The intended "more anatomical detail" never reaches the graph. A non-trivial spatial encoder (e.g., learned 3×3 → 1×1 with conv, or token-per-cell flatten) is needed to actually use stage-3.

2. **Comparison-only pooling collapses strict-FAR.** Excluding the global node from `SymmetricPairPooler` removes the most identity-correlated feature, but in this small model it was also the strongest separator at the extreme score tails. The graph still receives the global as context, so the global *signal* isn't lost — but the pooled fusion vector no longer carries it directly, and the classifier (which only sees the pooler output) loses the ability to push hard positives past the threshold needed for FAR=0.001.

3. **Relation aux λ=0.05 with stage-3 + comparison-only-pooling is mis-tuned.** On M12 the same λ helps because the M12 head still carries the global tokens directly. Here it works against the simultaneous identity-removal pressure: the relation head needs the global token to be informative (it's the input to `relation_head`), while the pooler now ignores it. Two opposing pressures compete on the same global tokens.

## Implications

- The "stage-3 + comparison-only + rel-aux" bundle was the highest-EV combined remediation; landing at 0.8526 says the graph-transformer + 8-landmark architecture on pre-aligned FIW cannot reach M12 R011 territory under any combination of the levers we've tried.
- **M13 line is stopped again.** Per the plan, no R003 launches.
- The 14×14 finding (point 1 above) suggests a future architectural revision *could* work, but it would no longer be the "minimal landmark graph" the line was meant to validate.
- Best.pt remains in `output/002/checkpoints/best.pt` (339 MB, ep 44, Val AUC 0.8952) in case future work wants to start from a partially-trained R002 seed.

## Disk-full incident (2026-05-28 ~ 08:00)

The training process crashed during `torch.save()` of the ep-85 periodic checkpoint. Root cause: 17 `epoch_{5..85}.pt` snapshots × 339 MB each + `best.pt` filled the disk (100 % used, 4.3 MB free). Best.pt at ep 44 was already written before the crash. Recovery:

1. Deleted all `epoch_*.pt` files (~5.7 GB freed).
2. Patched `best.pt` to add `model_config` + `protocol` (training crashed before its post-train save).
3. Ran `test.py --threshold 0.1` manually.

Lesson for future M13/M12 runs on this disk: the trainer's `save_every=5` policy + `~340 MB` checkpoints + `100 epochs` schedule = ~6.5 GB just in periodic snapshots. Should either reduce `save_every` to 25, prune older snapshots in the trainer, or move outputs to a non-system disk.
