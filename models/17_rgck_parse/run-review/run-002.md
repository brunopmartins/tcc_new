# Run 002 — Planning doc — M17 PG-RGCK (parsing boxes @ finer 224 map)

**Status:** Prepared 2026-06-16, not yet launched. Same code as R001 (no new
code); only `IMG_SIZE` changes. Launch after R001.

R002 = R001 (parsing-guided per-image boxes, M12 R011 recipe) with the conv body
at the native **IMG_SIZE=224 → a 14×14 feature map** instead of R001's
`160 → 10×10`. Same box cache (boxes live in the 224 frame; `src_coord_size=224`
normalises, so the SAME `.npz` works at any input size — no re-parsing).

## Why a finer map

Adaptive per-face boxes only pay off if the feature map can resolve the
per-region differences they encode. At `160 → 10×10`, each region box maps to
1–3 grid cells — the coarse-ROI problem M13/R013 documented, where "the
anatomical detail is already smoothed away by the time we ROI-pool." At
`224 → 14×14` (4× the cells), an adaptive box that is, say, 15% tighter or
shifted on one face vs another actually lands on different feature cells. R002
gives the geometry room to bite; if R001 is flat but R002 gains, the bottleneck
was map resolution, not the idea.

## What to watch

1. **R002 vs R001 vs M15 R001 (and M15 @224 if available).** The clean read:
   does the adaptive-box effect grow as the map gets finer? Compare to a fixed-box
   run at the SAME 224 (a dummy-cache R002, or M15 @224) to isolate boxes from
   resolution.
2. **VRAM/time:** 224 is heavier than 160; watch for OOM and per-epoch time.
3. Same parser-fallback and per-relation / low-FAR diagnostics as R001.

## Decision rules

- **R002 > R001 and > fixed-box@224** → adaptive geometry helps once the map
  resolves it; this is the headline. CV.
- **R002 ≈ fixed-box@224** → boxes don't help even at high resolution; combined
  with R001 this is a solid negative for the box-source axis on pre-aligned FIW.
- **R002 OOM/too slow** → fall back to IMG_SIZE=192, or keep R001 @160 as the
  primary and note the resolution limit.
