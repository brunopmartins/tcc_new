## Context

After R001-R006 plateaued at Test ROC AUC ≈ 0.81 with Val→Test gap ≈ -0.10
across all interventions (regularization, class balancing, partial unfreeze,
full unfreeze, heavy contrastive), a literature review (`IMPROVEMENT_OPTIONS.md`)
identified the architectural diagnostic:

> DINOv2 features are general-purpose self-supervised; kinship verification
> SOTA uses **face-specific pretrained backbones** (ArcFace, AdaFace) precisely
> because their embeddings are pre-organized for face-pair discrimination.

Three published recipes concentrate around this insight:

1. **FaCoR (Su et al. 2023, arXiv:2304.04546)** — uses AdaFace as backbone for
   kinship verification, achieves Test Acc = 82% on FIW.
2. **FRoundation (Boutros et al. 2024-2025, arXiv:2410.23831)** — establishes
   the canonical recipe for adapting DINOv2 to face recognition: LoRA r=16,
   CosFace loss, LR=1e-4. Reaches 87-91% verification accuracy on standard
   face benchmarks.
3. **Shadrikov 2020 (arXiv:2006.11739)** — explicitly notes that ArcFace
   pretrained "performs much better than other competitors for kinship
   recognition".

R007 takes a different path: instead of replacing DINOv2 with a face-specific
backbone (which would just replicate FaCoR/Shadrikov's recipe), **combine
both backbones in parallel and let the model use whichever features are
relevant per image region**.

The novelty lies in:
- **Combining a self-supervised foundation model with a kinship-supervised
  ViT as complementary backbones.** No published kinship paper does this.
- **Using our own M02 R031 checkpoint as the face-specific backbone**
  (instead of ArcFace pretrained on MS-Celeb-1M). The M02 ViT is already
  kinship-tuned via supervised contrastive on FIW — effectively a
  "task-specific pretrained encoder", which is itself a slightly different
  proposition than face-identity pretraining.
- **Differential cross-attention as the fusion mechanism.** Ye et al. 2024
  introduced DiffAttn for LLMs; we already use it inter-face in M05; R007
  extends it to intra-face fusion of two complementary backbones.

---

## Hypothesis

H0 (null): combining DINOv2 + M02-trained ViT does not improve over
M05 R001 (Test AUC = 0.806). The two backbones encode redundant information
at the level relevant to kinship.

H1 (target): the hybrid backbone reaches Test AUC ≥ 0.83. The DINOv2
stream contributes general visual richness (texture, fine detail across
demographics), the M02 stream contributes face-pair discrimination already
optimized for kinship. Together, the model exploits both.

H2 (stretch): Test AUC ≥ 0.85, matching or exceeding M02 R031. Possible
because DINOv2 features compensate for M02 R031's failure modes (extreme
age gap, atypical demographics) — the M02 ViT alone caps at AUC=0.85 in
FIW; adding DINOv2 features should boost the hardest pairs.

H3 (worst case): the M02 backbone dominates the fusion (because its
features are already kinship-discriminative) and DINOv2 contribution is
negligible. The model reduces to "M02 + extra cross-attention layers" —
similar to M02 with marginally more capacity, similar AUC.

The Val→Test gap is the secondary outcome. Three runs confirmed the gap is
approximately -0.10 with M05's standard architecture. R007's hybrid setup
might reduce the gap (closer to M02's -0.03) **if the M02 backbone's
generalization properties carry over** — that is, if part of M02's smaller
gap was due to its face-specific features generalizing better across
families, not just the loss recipe.

## Expected Outcomes (pre-experiment)

### If H1 holds (Test AUC ≥ 0.83)
**Validates the hybrid hypothesis.** The contribution becomes "DINOv2 +
task-specific ViT fusion as a unique architecture for kinship". Followup R008
would explore variations (different M02 checkpoints, different intra-face
attention layer counts, ablations of each backbone alone).

### If H2 holds (Test AUC ≥ 0.85)
**Beats M02 R031.** Strongest possible outcome. The fusion explains the
delta — neither backbone alone reached this number. Becomes a publishable
contribution.

### If H3 holds (M02 backbone dominates)
**Diagnose via ablation:** rerun with DINOv2 backbone disabled (zeros) to
measure pure M02 contribution. If that ≈ R007's number, the DINOv2 stream
is dead weight. Should then try inverting the dominance via gating layer
weights or by initializing the dinov2_proj close to identity.

### If H0 holds (Test AUC ≈ R001)
**Negative result confirms the structural hypothesis.** The Val→Test gap
of -0.10 is genuinely the RFIW Track-I split, not the model. M05 ceiling
is at AUC=0.81 for any frozen-backbone configuration. Strong motivation to
try (a) full fine-tuning of the hybrid (R008 = R007 + unfreeze), (b)
replacing DINOv2 with another foundation model, or (c) accepting M05's
ceiling and pivoting to a different model architecture (M07).

### Why it failed

The M02 backbone, when extracted in isolation and combined with DINOv2 via a small fusion module, **does not carry over its full M02-R031 performance**. M02's per-relation excellence (88-96% across grandparent classes) came from the **trained-together ecosystem**:

- ViT-B/16 backbone (from ImageNet, fine-tuned during training)
- Bidirectional FaCoR cross-attention (trained jointly with backbone)
- Channel attention module (trained jointly)
- Final projection + supervised contrastive loss (margin=0.3)
- Threshold = 0.90 (very high, paired with sharply polarized score distribution)

R007 reused only the **ViT weights** of M02. The cross-attention design, the loss, the threshold calibration — all gone. The face-discriminative features that M02-ViT encodes are still useful for some signal (Test AUC = 0.810 is not random), but they're now consumed by an architecture (M05's DiffAttn) that wasn't co-designed with them.

Conclusion: **face-specific pretraining alone is not enough** — the head architecture, loss, and pretraining must compose. This is consistent with the kinship literature (Shadrikov 2020, FaCoRNet) where backbone choice + custom head are inseparable.

### Bug found and fixed

The first run of `test.py` on R007 produced **Test AUC = 0.504 (random)** because `test.py::build_model_from_checkpoint` always instantiated `DINOv2LoRAKinship`, ignoring the hybrid case. With `strict=False` in `load_state_dict`, the `face_vit.*` and `intra_face_layers.*` weights were silently discarded.

Fixed in both [test.py](../AMD/test.py) and [evaluate.py](../AMD/evaluate.py): now detect hybrid architecture by inspecting state-dict prefixes (`face_vit.`, `intra_face_layers.`) and instantiate `DINOv2HybridKinship` accordingly. Re-running with the fix gave the real Test AUC = 0.810 reported above.

### Implications

1. **The recipe gap (FRoundation) is more important than the backbone hybrid.** R007 confirms that combining backbones doesn't help if the loss/head/training regime is wrong.
2. **M07 should pivot away from "frozen backbone + adapter" and toward "ArcFace pretrained + full fine-tune"** if pursued. R007 confirmed the limit of frozen-backbone variants.
3. **R001 remains the Pareto-optimal M05 checkpoint** for TAR@FAR=0.01. R005 for AUC. R007 for high-precision recall trade-off.
