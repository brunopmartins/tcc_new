## Context

R001 e R002 deixaram tres pistas convergentes para a falha do M05 em classes de avo:
1. R001 → gfgs/gmgd em 39.8%/40.7% (proximo de chance)
2. R002 (regularizacao mais forte) → gap val→teste **nao fechou** (-0.106 vs -0.105 R001), descartou overfitting puro
3. Diagnostico de starvation revelou que classes de avo tem 1.6-2.1% do treino (vs 15-16% das classes de pais)

R003 testa a hipotese de que **starvation de dados e o gargalo principal**. Se for, dar 4-6× mais updates por epoch a essas classes deveria mover seu desempenho para perto do que classes mais frequentes alcancam.

A intervencao: substituir o sampler aleatorio por `WeightedRandomSampler` que entrega 50% de negativos + 50% de positivos divididos igualmente entre as 11 relacoes do FIW. Sampler verificado empiricamente — gfgs vai de 1.03% para 4.54% das amostras por epoch.

Todos os outros hyperparams identicos a R001 (LR=3e-4, dropout=0.1, λ_rel=0.2). Ablation limpa.

## Analysis

### Hypothesis test result: REJECTED

A hipotese era: data starvation explica a falha em classes de avo → mais updates resolvem. R003 deu 4-6× mais updates verificadamente. Resultado: classes de avo regrediram. **A starvation nao era o gargalo.**

### Tres interpretacoes possiveis (em ordem de probabilidade)

**1. Gargalo arquitetonico — features fixas (mais provavel).** DINOv2 frozen extrai features que ja nao distinguem classes de avo bem. LoRA + heads so podem combinar features que existem. Mais updates de gradiente sobre features fixas que nao codificam o sinal certo nao ajuda — esta tentando extrair informacao que nao esta no input. Esta leitura e reforcada por R002 (regularizacao tambem nao ajudou) e converge para a necessidade de descongelar o backbone (R004).

**2. Threshold drift como artefato.** Threshold drifou de 0.10 (R001) → 0.30 (R002) → 0.50 (R003). Per-relation accuracy depende de threshold. **Parte** da queda em R003 e artefato — sob threshold mais alto, recall cai em todas as classes. Mas TAR@FAR=0.01 caiu de 0.152 para 0.098 (threshold-independent), entao o ranking real tambem piorou na regiao de alta precisao.

**3. Confound de pos/neg ratio.** Implementacao foi 50/50 pos/neg + balance entre 11 classes positivas. R001 tinha 33/67 pos/neg natural. A mudanca para 50/50 confunde dois experimentos em um. Nao explica o resultado isoladamente, mas adiciona ruido. Versao mais limpa: preservar 2:1 e so rebalancear dentro dos positivos.

### Convergencia das tres runs do M05

| Run | Hipotese testada | Intervencao | Resultado |
|-----|-----------------|-------------|-----------|
| R001 | — (defaults) | — | Test AUC 0.806, val→teste gap -0.105, gfgs 39.8% |
| R002 | "Gap = overfitting" | LR/2 + dropout 2× + lora_dropout 0.1 + λ_rel 0.4 | **Rejeitada.** Gap unchanged (-0.106) |
| R003 | "Avos = starvation" | Stratified sampler 4-6× mais updates | **Rejeitada.** Avos regrediram |

Tres hipoteses obvias eliminadas. A unica direcao remanescente eh **mover o teto arquitetonico** descongelando o backbone DINOv2.

### TAR@FAR observation

R001 tem TAR@FAR=0.01 = 0.152 — o melhor do projeto inteiro, supera M02/M03 (~0.13) e M06 (~0.06). Cada intervencao posterior em M05 piorou esse numero. **R001 deve ser tratado como o checkpoint canonico do M05 para uso de alta precisao**, mesmo que outras runs tenham AUC similar.
