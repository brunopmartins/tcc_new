# Run 012 — Planning doc — Consistency loss + extended stage-3 unfreeze

**Status:** Implementation complete (2026-05-27). Not yet launched.
GPU is currently occupied by M13 R002 (output/002/).

R012 = R011 stack (R006 symmetric forward + R007 differential LR +
R009 comparison-only fusion + role-matched hard negatives at 30 %) plus:

- **Cosine-consistency loss between `fusion_AB` and `fusion_BA`**, λ=0.05.
- **Extended stage-3 unfreeze**: also unfreeze `body[43:46]` on top of
  the existing `body[46:49] + output_layer`.

## Hypothesis

The 5-fold CV of R006, R010, and R011 all landed inside the same narrow
AUC band (CV means 0.8733, 0.8739, 0.8761; CV σ ≈ 0.003–0.004). Three
substantially different architectural interventions converged on the
same ceiling. Two complementary mechanisms could plausibly break it:

1. **Stronger order-invariance.** R006's Option-B BCE forces
   `f(A,B) ≈ f(B,A)` only at the final logit. Each direction's
   `cross_region` adapter is free to learn *different* fusion vectors
   that just happen to average out at the BCE level. A direct cosine
   penalty on `(fusion_AB, fusion_BA)` forbids that and propagates the
   constraint back into the cross-region adapter weights.
2. **More backbone capacity for the harder constraint.** R001–R011 all
   unfroze the *same* 3 BasicBlockIR blocks (`body[46:49]`) plus the
   FC `output_layer`. Adding `body[43:46]` (the tail of stage 3) gives
   the backbone ~3.5 M extra trainable parameters to absorb the
   consistency objective without collapsing the comparison features.

The two changes are bundled because the consistency loss is what
*motivates* the extra capacity. Without it, the extra unfreeze is just
the L2-SP / capacity-reallocation lever (R007/R008 territory), and we
already know that doesn't help on its own.

## Implementation

1. `model.py`:
   - `_forward_head` now returns a 7-tuple ending with the pre-classifier
     `fusion` tensor.
   - In `forward(...)` (symmetric mode), `fusion_ab` and `fusion_ba` are
     stored in `sym_extras`.
   - New `unfreeze_extra_stage3_tail` flag on `RGCKNet.__init__` and
     `build_rgck_net(...)`; when set together with `unfreeze_last_stage`,
     also flips `requires_grad=True` on `backbone.body[43:46]`.

2. `AMD/train.py`:
   - New CLI flags `--consistency_weight` (default 0.0) and
     `--unfreeze_extra_stage3_tail`.
   - `RGCKBCELoss` accepts `consistency_weight` and adds
     `λ * (1 - cos(fusion_AB, fusion_BA)).mean()` to the total loss
     when the model is in symmetric mode and the term is enabled.
   - `build_differential_lr_optimizer_and_scheduler` includes a new
     `backbone_stage3_tail` param group (sharing `lr_stage4`).
   - The new flags are persisted in the checkpoint's `model_config`.

3. `AMD/test.py` / `AMD/cv_ensemble.py`:
   - Both read `unfreeze_extra_stage3_tail` from `model_config` when
     rebuilding for inference.

4. `AMD/run_pipeline.sh`:
   - New env vars `UNFREEZE_EXTRA_STAGE3_TAIL` and `CONSISTENCY_WEIGHT`.
   - Banner reports both; TRAIN_ARGS forwards both to `train.py`.

5. `AMD/run_r012.sh`:
   - One-shot wrapper that sets the R012 recipe.

## Smoke tests (passed)

- Model construction with both new flags produces a 7-tuple whose
  `sym_extras` carries `fusion_ab`/`fusion_ba` at shape `(B, 1035)`
  (the comparison-only fusion vector). Trainable params 34.6 M /
  70.2 M (49.24 %), up from R011's 31.0 M (44.20 %).
- `RGCKBCELoss(consistency_weight=0.05)` produces a finite loss on
  random inputs with relation_idx masking.

## Decision rules (for when R012 is launched)

- **R012 Test AUC ≥ 0.880** (single run): clear evidence the ceiling
  is broken. Run 5-fold CV next, treat as a new headline candidate.
- **0.876 ≤ Test AUC < 0.880**: in the ceiling band; check whether
  low-FAR moved meaningfully above R011's 0.0677 ± 0.0147. If yes,
  CV the recipe. If no, file R012 as "neutral" alongside R007/R008.
- **Test AUC < 0.876**: the consistency loss is restricting the model
  too aggressively. Sweep `CONSISTENCY_WEIGHT ∈ {0.01, 0.02}` before
  abandoning the line.
- **Val→test gap widens > 0.05**: extra-stage3 unfreeze is overfitting.
  Drop `UNFREEZE_EXTRA_STAGE3_TAIL=0` and retry with consistency only.

## Risks and notes

- The consistency loss only fires in symmetric mode. The wrapper
  enforces `SYMMETRIC_FORWARD=1`; the trainer warns if the user
  enables consistency without symmetric.
- Adding ~3.5 M trainable parameters reduces the regularising effect
  of the still-frozen stages 1–2. If R011's noise floor was capacity-
  saturated this gives room to move; if it was *generalisation*-
  saturated this will widen val→test.
- `consistency_weight=0.05` matches the relation-aux weight by design.
  If both auxiliary terms dominate BCE, the model can drift to "make
  fusion symmetric and predict the right relation" without solving the
  kinship task. Inspect train BCE separately during epochs 1–5.
