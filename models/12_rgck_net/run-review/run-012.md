## Context

The 5-fold CV of R006, R010 and R011 all landed inside the same narrow
AUC band (CV means 0.8733, 0.8739, 0.8761; CV σ ≈ 0.003–0.004). Three
substantially different architectural interventions converged on the
same ceiling. The only thing that cleanly broke it was the R011 CV
**ensemble** (0.8839) — averaging 5 models, not a single model.

R012 is the last single-model lever in the shortcut-removal family.
Two complementary mechanisms are bundled because the second motivates
the first:

1. **Stronger order-invariance.** R006's Option-B BCE forces
   `f(A,B) ≈ f(B,A)` only at the final logit. Each direction's
   `cross_region` adapter is free to learn *different* fusion vectors
   that just happen to average out at the BCE level — direction-specific
   shortcuts that cancel only on average. A direct cosine penalty on
   `(fusion_AB, fusion_BA)` forbids that and propagates the constraint
   back into the cross-region adapter weights.
2. **More backbone capacity for the harder constraint.** R001–R011 all
   unfroze the same 3 BasicBlockIR blocks (`body[46:49]`) + the FC
   `output_layer`. Adding `body[43:46]` (stage-3 tail) gives the
   backbone ~3.5 M extra trainable parameters to absorb the consistency
   objective without collapsing the comparison features. Without the
   consistency loss the extra unfreeze is just the R007/R008
   capacity-reallocation lever, which we already know is neutral on its
   own — so the two changes are tested together.

**Hypothesis:** the AUC ceiling at 0.876 ± 0.003 is partly because
`cross_region(A,B)` and `cross_region(B,A)` still produce meaningfully
different fusion vectors even under symmetric BCE. An explicit
consistency penalty + the capacity to satisfy it should push past 0.880.

motivates the extra capacity). The loss becomes:

```
L = 0.5·(BCE_AB + BCE_BA) + 0.05·avg(CE_rel_AB, CE_rel_BA)|pos

```

## Analysis

### What worked

- **TAR@FAR=0.001 = 0.0878 — project-best single-run strict-FAR.**
  Above R011 single (0.0751) and the R011 CV ensemble (0.0801). If this
  survives CV it makes R012 the low-FAR headline. Strict-FAR is the
  operationally meaningful metric for kinship verification in a
  low-base-rate setting, so this is the result that matters.
- **Val→test gap = -0.026, unchanged from R006.** The +3.54 M stage-3
  parameters did **not** widen the generalisation gap — the consistency
  loss + frozen stages 1–2 held it. The decision rule's overfitting
  tripwire (gap > 0.05) did not fire. The capacity bet was safe.
- **Test AP = 0.8636** and **TAR@FAR=0.1 = 0.6105**, both above every
  prior single model except the R011 ensemble. Aggregate ranking quality
  is at least as good as R006/R011.
- **Clean BCE descent** confirms the two 0.05 auxiliary terms did not
  starve the primary objective.

### What is uncertain (the decisive caveat)

- **AUC 0.8813 is inside the noise floor.** It is +0.0074 over R010 CV
  mean and +0.0052 over R011 CV mean — both below the project's
  ±0.008 single-run decision threshold. R011's single run (0.8825)
  looked like a break too and collapsed to a CV mean of 0.8761; R012
  could be the same upper-tail draw. **On AUC alone this is not a
  confirmed break, and it does not beat the 0.8839 ensemble headline.**
- **Precision did not improve** (73.09 %, essentially tied with R010/R011
  CV means). The gain is concentrated in strict-FAR ranking, not the
  operating-point precision. The recall-heavy threshold 0.400 means the
  deployed precision is among the lower M12 values.

### Verdict against the pre-registered decision rules ([planning section below])

| Rule | Threshold | R012 | Verdict |
|---|---|---|---|
| AUC ≥ 0.880 → CV, headline candidate | ≥ 0.880 | 0.8813 | **✓ met — CV launched** |
| low-FAR above R011's 0.0677 ± 0.0147 | rise | 0.0878 (+1.4σ) | **✓ comfortably met** |
| Val→test gap > 0.05 → drop extra unfreeze | < 0.05 | -0.026 | **✓ no overfit** |
| AUC < 0.876 → sweep consistency 0.01/0.02 | n/a | — | not triggered |

The AUC ≥ 0.880 rule fired, so the prescribed next step is the 5-fold
CV (launched). The secondary low-FAR criterion is met with margin,
which is the more promising signal.

### Mechanism

R012 adds a **fourth + fifth** intervention on top of the R005–R011
taxonomy:

| Intervention | Targets | Mechanism |
|---|---|---|
| R006 symmetric forward | aggregate AUC | order-invariance at the logit |
| R009 comparison-only fusion | strict-FAR | removes gA/gB identity leakage |
| R011 role-matched hard negs | operating curve + specificity | closes the easy-negative gap |
| **R012 consistency loss** | **fusion-level order-invariance** | forbids per-direction shortcuts that cancel only on average |
| **R012 stage-3 tail unfreeze** | **capacity for the constraint** | room to satisfy consistency without collapsing comparison features |

The single-run signal is consistent with the hypothesis: the consistency
penalty pushed strict-FAR (where direction-shortcut cancellation most
distorts the score tail) without widening the generalisation gap. But
the aggregate-AUC ceiling held within noise — exactly the ambiguous
outcome the CV is designed to resolve.

## Implementation (reference)

1. `model.py`:
   - `_forward_head` returns a 7-tuple ending with the pre-classifier
     `fusion` tensor; `forward(...)` stores `fusion_ab`/`fusion_ba` in
     `sym_extras` (shape `(B, 1035)` under comparison-only fusion).
   - `unfreeze_extra_stage3_tail` flag on `RGCKNet.__init__` /
     `build_rgck_net(...)`; with `unfreeze_last_stage` it also flips
     `requires_grad=True` on `backbone.body[43:46]`.
2. `AMD/train.py`:
   - CLI `--consistency_weight` (default 0.0) and
     `--unfreeze_extra_stage3_tail`.
   - `RGCKBCELoss` adds `λ·(1 − cos(fusion_AB, fusion_BA)).mean()` in
     symmetric mode.
   - `build_differential_lr_optimizer_and_scheduler` adds a
     `backbone_stage3_tail` group (shares `lr_stage4`).
   - Both flags persisted in `model_config`.
3. `AMD/test.py` / `AMD/cv_ensemble.py`: read
   `unfreeze_extra_stage3_tail` from `model_config` when rebuilding.
4. `AMD/run_pipeline.sh`: env vars `UNFREEZE_EXTRA_STAGE3_TAIL`,
   `CONSISTENCY_WEIGHT`; forwarded to `train.py`.
5. `AMD/run_r012.sh`: one-shot single-run wrapper.
6. `AMD/cv_runner_r012.sh`: 5-fold CV runner (R012 recipe + 15-min
   cooldowns + initial GPU-warm cooldown).

**Smoke tests (pre-launch, passed):** model construction with both flags
produces a 7-tuple whose `sym_extras` carries `fusion_ab`/`fusion_ba` at
`(B, 1035)`; trainable params 34.6 M / 70.2 M (49.24 %);
`RGCKBCELoss(consistency_weight=0.05)` returns a finite loss with
relation_idx masking.
