This run is the **fourth attempt** at M11 R001. The first three (v1/v2/v3)

## Context

M10 and M02 already implement the FaCoR cross-attention module. M09 adds
inside-the-backbone multi-stage cross-attention. M11 was designed to
test the *training-recipe side* of the FaCoRNet paper — specifically:

- attention-driven dynamic temperature contrastive loss (`relation_guided`)
- `relation_matched` hard negatives (positives' partners drawn from
  different families but matching role)

The premise: M09 R001 hit Test AUC 0.7982 with random negatives. If
FaCoRNet's hard-negative mining gives the model a better discriminative
signal, M11 should beat M09 R001 on test families.

It did not.

## Failed attempts (v1 / v2 / v3) — bug discovery

| Attempt | Loss + temp                        | ep 1 outcome                                |
|---------|-----------------------------------|---------------------------------------------|
| v1      | `relation_guided`, temp 0.07      | train 0.80→0.02 in 1 ep; per-rel 1.000 uniform; Val Acc trapped at 50 % |
| v2      | `relation_guided`, temp 0.3       | slower collapse; same degenerate state      |
| v3      | `cosine_contrastive`, temp 0.3    | same degenerate state                       |

The common signature was **predict-all-kin**: every (kin) class shows
100 % accuracy because the model classifies *every* face pair as kin.
The non-kin classes (which dominate the val set) all fail, dragging the
balanced Val Accuracy to exactly 50 %.

The user's audit ([discussao.md](../discussao.md)) identified the root
cause: **`cosine_contrastive` and `relation_guided` losses ignore the
batch labels.** The forward signatures accept `labels` (or attention)
but never use them in the loss computation. Every `(emb1_i, emb2_i)`
correspondence is treated as a positive in the InfoNCE numerator,
regardless of whether the dataset labelled the pair kin or non-kin.

Confirmed via direct inspection of [models/shared/losses.py:74](../../shared/losses.py#L74):

> `labels: Optional labels (not used in standard InfoNCE)`

Combined with `relation_matched` hard negatives, this trained the model
to *pull non-kin pairs together*, hence the all-kin collapse.

This is a **pre-existing project bug**, not specific to M11. M10 R002
showed the same instability with `cosine_contrastive` on AdaFace (val
AUC oscillating 0.57-0.71, threshold flipping between 0.1 and 0.9). At
the time we attributed it to "AdaFace identity-cluster prior conflicting
with contrastive loss" — the actual cause was simpler and more serious:
the loss ignores labels. Same code path.

The v1/v2/v3 attempts produced no useful checkpoints; their outputs were
cleaned up. Only v4 survived to a test phase.

## Analysis

### What worked

- **The bug fix worked.** v4 trained cleanly and achieved the highest
  Val AUC of any AdaFace-based model in the project (0.8987 > M09
  R001's 0.8919).
- **Unlock magnitude (+0.127) was *larger* than M09 R001's** (+0.111).
  The hard-negative signal *during training* was stronger.
- **`gfgd` and `gfgs` improved** (+6.1 to +6.5 pp). The two grandfather
  classes that historically struggled with random negs benefited from
  the harder discriminative signal.

### What failed

- **Test AUC regressed -0.028 vs M09 R001** (0.7707 vs 0.7982). The
  *higher* Val AUC did not translate to a higher Test AUC — quite the
  opposite.
- **Val→test gap widened to -0.128** (M09 R001: -0.094, M09 R002:
  -0.107). M11 v4 has the *worst* generalisation of any successful
  AdaFace run.
- **Sibling classes regressed catastrophically** (sibs -14.5 pp, ss
  -9.0 pp, bb -7.6 pp). The relation-matched negatives drew from same-
  role positive pools (e.g., a `sibs` negative is a *sibs* pair from
  a different family). The model learned to over-discriminate on
  sibling-like pairs, hurting test sibling accuracy.
- **`gmgd` and `gmgs` regressed** (-8 to -10 pp) while `gfgd`/`gfgs`
  improved. Asymmetric effect across grandparent classes — possibly
  data-distribution artifact (smaller N magnifies sampler effects).

### Hypothesis test result

**"FaCoRNet's `relation_matched` hard negatives improve Test AUC on
top of M09's stack" — REJECTED.** Net Test AUC -0.028, gap widened
-0.034.

This is the **second M11 result that confirms a consistent pattern**:
sophisticating M09 R001's training distribution (via balanced sampling
in M09 R002, hard negatives here in M11 R001) raises Val AUC but
*lowers* Test AUC. The model becomes better at the val family pool
(which shares train families) and worse at held-out test families.

### Likely root cause

`relation_matched` negatives are *too similar to positives in the val
pool*. The sampler picks a "sibs" negative from another sibs pair in
the same train family pool. Validation pairs come from the same family
pool — so the model effectively trains on val-family-distribution-aware
negatives. This boosts Val AUC. But test families are completely held
out, so the val-pool calibration doesn't transfer.

This is structurally similar to M09 R002's balanced sampler issue:
both interventions trained the model on a *harder version of the
val-pool distribution*, raising Val AUC but not Test AUC.

### Implication

The M09 R001 stack (AdaFace IR-101 full FT + multistage cross-attn +
BCE classifier head + random negatives) appears to be the AdaFace-based
local maximum on our FIW pipeline. Two independent FaCoRNet-inspired
interventions failed to improve over it. The remaining unexplored
directions:

1. **Different backbone** — M02 R031's 0.850 still leads, suggesting
   AdaFace's identity-cluster prior is the limiting factor. ViT-B/16
   ImageNet stays competitive.
2. **Architectural changes that reduce backbone influence** — region
   tokens (M12 RGCK-Net Phase 1, frozen backbone), retrieval-based
   approaches (M06).
3. **Faithful FaCoRNet implementation** — positives-only batches with
   implicit negatives via InfoNCE denominator + paper-faithful Rel-Guide
   `M(beta)/s` with s=500. Requires loss rewrite per Caminho A of
   `discussao.md`.
