# Run 001 — Planning doc — M15 hi-res ROI-Align tokenizer (no 112 limit)

**Status:** Implemented 2026-06-10, not yet launched. Smoke-tested on CPU
(random init): builds in both global-token modes, tokens `(B, 5, 512)`,
forward returns the 7-tuple in symmetric mode, trainable params
**31,036,173 / 70,222,029 (44.20 %) — identical to M12 R011**.

M15 R001 = the **M12 R011 champion recipe** with the region tokenizer swapped
for the **high-resolution ROI-Align** tokenizer. Everything else (head, loss,
backbone unfreeze surface, data recipe, seed) is identical to R011, so the run
is a clean single-variable ablation.

## Hypothesis

The M12 single-model AUC ceiling (R006/R010/R011/R012 CV means all in
0.873–0.879) is set in part by **how region tokens are produced**, not only by
the frozen representation:

- M12 `FixedPartition` squashes each anatomical box to 112×112 and re-runs
  AdaFace — feeding the backbone **out-of-distribution crops** (a stretched
  eye/mouth strip is nothing like the whole aligned face AdaFace was trained
  on). Stages 1–3 are frozen, so they process these crops with face-tuned
  filters that don't apply.
- M12 R013 `ROIAlign` fixed the OOD-crop problem but ran the body at **112 →
  7×7**, so each region is pooled from 1–3 cells of a coarse 7×7 grid (the
  M13 coarse-ROI failure mode).

**Claim:** the 112 constraint is only AdaFace's FC head (`Linear(512·7·7, 512)`).
The conv body is resolution-agnostic. Running it at 224 (→ 14×14 map) and
ROI-Aligning each region to the 7×7 grid the FC needs gives **in-distribution,
undistorted region tokens with 4× the spatial detail** of R013 — which should
lift the per-region signal (especially the anatomically-localised classes) and
possibly the aggregate ceiling.

`global_token_mode=exact` holds the global token identical to M12/B0 (genuine
112 AdaFace embedding), so any movement vs R011 is attributable to the
**anatomical region tokens** alone.

## Changes from M12 R011

| Component | M12 R011 | **M15 R001** |
|---|---|---|
| Region tokenizer | `FixedPartition` (crop→112×112→AdaFace ×5) | **`HiResROIAlign`** (1 body pass @224 → 14×14 map → ROI-Align→7×7→FC) |
| Region tokens | OOD stretched crops | in-distribution, undistorted, 4× detail |
| Backbone passes / face | 5 (one per region @112) | 2 (224 spatial map + 112 global FC) |
| Global token | 112 AdaFace embedding | 112 AdaFace embedding (`exact`, identical) |
| Trainable params | 31,036,173 (44.20 %) | 31,036,173 (44.20 %) — identical |
| All other knobs | — | identical to R011 |

## What to watch (the real signal)

The M12 numbers are taken at the **Val-AUC peak checkpoint (≈ep 3)**, before the
memorisation phase. So watch two things:

1. **Peak Val AUC** — does it rise above R011's ~0.904 / R012's ~0.907? A higher
   peak (not just a later one) is the sign the finer region tokens add real
   discriminative signal.
2. **Per-relation accuracy on anatomically-localised classes** — the hypothesis
   predicts the biggest gains where region detail matters; siblings/parent-child
   should hold and the eyes/nose/mouth-driven separation should sharpen. If only
   the global signal carries the model (per-region cosines flat), the tokenizer
   change isn't helping.
3. **Val→test gap** — should stay tight (~-0.02 to -0.03) like R011; if it
   widens, the finer tokens are being memorised.

## Decision rules

- **Test AUC ≥ 0.884 (single run)**: clears the R011 CV-ensemble headline as a
  *single model* — strong result. Launch 5-fold CV immediately.
- **0.876 ≤ Test AUC < 0.884**: in the ceiling band. Check (a) peak Val AUC vs
  R011/R012 and (b) per-region accuracy. If either moved meaningfully, CV the
  recipe; the contribution may be per-class / interpretability rather than
  aggregate AUC.
- **Test AUC < 0.876**: the hi-res tokenizer didn't help on aggregate. Before
  abandoning, try the two cheap variants below.
- **Val→test gap > 0.05**: the finer tokens are overfitting; report it (it would
  itself be an informative negative — "more region detail → more memorisation").

## Cheap follow-up variants (if R001 lands in the band)

1. **`IMG_SIZE=336`** (→ 21×21 map, even finer ROIs). Watch VRAM
   (`BATCH_SIZE=4 GRAD_ACCUM=8` if needed). Tests whether more region detail
   keeps helping or saturates.
2. **`GLOBAL_TOKEN_MODE=roi`** — single backbone pass (cheaper). Tells us how
   much the exact-FC global is worth vs a fully feature-map-native model.
3. **CV-fold ensemble** of the 5 R001 folds (the move that took R011
   0.8761 → 0.8839). Realistic path to a headline number.

## Honest prior

The frozen stages 1–3 and the family-disjoint difficulty are unchanged, so the
most likely outcome is **AUC in the 0.876–0.879 band** with a cleaner per-region
profile and possibly better low-FAR. The clean win condition is a higher *peak*
Val AUC: that would mean the finer, in-distribution region tokens carry signal
the 112-squashed crops destroyed. Either way the result is publishable — it's
the controlled test of "does removing the 112 region constraint matter," which
the M12 line never ran (R013 stayed at 112 → 7×7).

## VRAM

Trainable 31.0 M (= R011). Body runs once @224 instead of 5× @112, so backbone
compute is comparable to M12. Fits 12 GB at batch 8 × grad-accum 4. Fallback:
batch 4 × grad-accum 8 (same effective 32).

## EXECUTION NOTE — 2026-06-11: @224 is GPU-blocked on this card

Launched at the planned `IMG_SIZE=224` and measured **steady 4.6 s/it (~10 h/epoch)**
— unviable. Root cause: the AdaFace IR-101 conv body at 224 produces feature-map
shapes for which MIOpen has no fast kernel on this gfx1031 / ROCm 5.7 setup
(~25× slower than the native 112). Not a code bug — the model builds and trains
correctly (31.0 M params, hi-res tokenizer confirmed); it's a hardware/library
limitation. (Same reason the M14-R002 LoRA@224 attempt and the R013@224 idea were
dropped.) Aborted at @224.

**Relaunched at `IMG_SIZE=160`** (→ 10×10 feature map, still finer than R013's
7×7 but with much smaller convs). `run_m15_r001.sh` made `IMG_SIZE`-overridable
(default 160). Measured **2.1 s/it (~4.9 h/epoch → ~70 h full run)** — only ~2×
faster than 224, still far slower than the native-112 lines (~0.18 s/it). So even
160 is MIOpen-slow here; the hi-res ROI direction is **hardware-bottlenecked at any
non-112 resolution** on this card. Running (pending result) but expensive.

Takeaway: the hi-res ROI idea is sound but **impractical on this GPU** — any
input > 112 hits slow conv kernels (224 ≈ 10 h/epoch, 160 ≈ 5 h/epoch vs ~5 min
at 112). Would need a better-supported GPU/library to evaluate fairly.

## Caveats & limitations (added 2026-06-11)

**No test result exists yet** — nothing can be concluded about the hi-res
hypothesis. State of the run:
- **@224 (the planned, headline config): never evaluated** — aborted on speed
  (~10 h/epoch). So the 14×14-grid version the hypothesis is actually about has
  **zero** datapoints.
- **@160: still on epoch 1** at the time of writing (~4.9 h/epoch); best.pt / test
  pending. When it lands it must get the same rigor as M16/M14 (report **Val AUC +
  val→test gap**, decompose train loss vs the 0.05·CE_rel term, N).

**Even a completed @160 is a weakened test of the hypothesis:**
1. **Resolution reduced from the plan.** The hypothesis is "finer region grid
   helps" and was specified at 224 (14×14). @160 gives only 10×10 — a partial,
   smaller-grid test; a null at @160 would not refute @224.
2. **Source faces are ~110 px.** Both 160 and 224 are **upsampled** from the
   native ~110 px FIW crops, so there is **no new pixel detail** — the only thing
   that changes is the feature-grid resolution for ROI pooling. A positive result
   would be attributable to the finer grid, not to "more detail"; a null is
   expected if the grid was not the bottleneck.
3. **N = 1**, single resolution, single seed, FixedPartition→ROI is the only delta
   vs R011 (good for isolation) but still one run.

### Verdict — POSITIVE on ranking (the one lever in the campaign that helped)

The finer 10×10 region grid clearly improved the **score ranking**:
- **AUC 0.8803** — top of the single-run band (~R012, within noise of R011 CV
  +0.004; below the R011 *ensemble* 0.8839). So **no clear aggregate-AUC break**.
- **AP 0.8760 and the whole TAR@FAR curve are the best in the project** — this
  single model **beats even the R011 CV-ensemble** on AP (+0.010), TAR@FAR=0.001
  (**0.099 vs 0.080**), 0.01 (0.291 vs 0.217), 0.1 (0.647 vs 0.606). These are
  **threshold-invariant**, so they are genuine ranking gains, not the thr-0.550
  operating point (which traded recall 0.818 for precision 0.770).

Mechanism reading: undistorted, finer-grid region tokens sharpen the **low-FAR
tail** (exactly where verification cares) and positive-class ranking (AP), even
though aggregate AUC is near the wall. This is the **first lever in the M14/M15/M16
campaign to beat R011** on anything outside noise.

### Caveats (unchanged + new)
- **Single run; best.pt = ep4 = the warmup boundary** (LR ramp ends ep5 — same
  transient-selection concern as M14). Manually finalised (thr 0.550).
- **@160, not the planned @224** (14×14) — this is the *weakened* proxy and it
  still won; @224 is GPU-blocked here but could be stronger.
- Source ~110 px ⇒ both upsampled; the win is the **finer feature grid**, not new
  pixel detail. N=1.

### Strongly recommended next
**CV M15 @160** (5 family-disjoint folds) — the AP/low-FAR gains are large enough
(AP +0.020, TAR@FAR=0.001 +0.032 vs R011 CV) that confirming them across folds
would make M15 a real project result (best low-FAR / AP model), and a CV-ensemble
(the move that took R011→0.8839 on AUC) could push AUC too. If a faster GPU
becomes available, run @224 for the full-strength test.
