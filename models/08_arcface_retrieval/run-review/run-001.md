## Context

Após 5 runs do Modelo 05 plateauando em Test AUC ≈ 0.81-0.82, a hipótese central
do Modelo 08 era simples: **trocar o encoder visual genérico (DINOv2/ViT) por
um encoder face-specialized (ArcFace) deveria destravar o teto.** Literatura
de kinship (FaCoRNet, Adv-Mix, etc) usa ArcFace IResNet-100 como backbone
padrão, e modelos de face recognition atingem >99.8% em LFW — features
qualitativamente superiores para qualquer tarefa que envolva rostos.

Arquitetura idêntica ao Modelo 06 R001 (frozen encoder + non-parametric
gallery + cross-attention over top-K retrieved supports + binary head +
relation head). Single change: encoder DINOv2/ViT-B → ArcFace IResNet-100.

Esta run usa imagens MTCNN-alinhadas (`datasets/FIW_aligned/`, 224×224
redimensionadas para 112×112, normalização ArcFace [-1, 1]).

## Analysis — Conclusão completa

### A hipótese central (anti-kinship) é CONFIRMADA pelo Test AUC

Test ROC-AUC = 0.693, o **pior resultado de teste do projeto**, **8.3 pp
abaixo** do mesmo arranjo arquitetônico com ImageNet ViT (M06 R001 = 0.776)
e **15.7 pp abaixo** do melhor modelo do projeto (M02 R031 = 0.850).

Três evidências independentes (3 weights ArcFace diferentes) convergem para
o mesmo padrão:

1. **ArcFace IResNet-50 / CASIA-FaceV5** (ep 1 abortado): Val AUC = 0.6826
2. **ArcFace IResNet-100 / Glint360K** (ep 1 abortado): Val AUC = 0.7619
3. **ArcFace IResNet-100 / MS1MV3** (full run): Val AUC peak = 0.7455,
   declinou para 0.6256, Test = 0.6931

Todas as três variantes ficam **abaixo do M06 R001 com ImageNet ViT genérico
frozen** (Val AUC ep 1 = 0.7820, Test = 0.776). Não é um problema de
qualidade do pretreino — é o **paradigma de treino ArcFace** que ativamente
suprime sinal genealógico em modo frozen.

ArcFace foi treinado para **separar identidades** com loss de margem angular.
Os datasets de pretreino (MS1MV2/MV3, Glint360K) catalogam pais, filhos e
irmãos como **identidades distintas**, e a loss explicitamente os **empurra
para regiões diferentes** do espaço de embedding. O sinal genealógico
(traços herdados) é precisamente o que esse treino é projetado para **negar**.

### A descoberta inesperada: dissociação entre recall per-classe e AUC global

Surpresa contraintuitiva: M08 R001 atinge recall **significativamente melhor
que M05 R001 nas classes raras de avo/avo** (gmgd +30.8 pp, gfgs +31.6 pp,
gfgd +23.9 pp). Como pode um modelo com pior AUC global ter melhor
per-classe em classes específicas?

**Reinterpretação mais nuançada da hipótese anti-kinship:**

- **ArcFace features são úteis para retrieval** (similaridade facial). A
  galeria de 33.207 pares ArcFace consegue achar parentes "que se parecem
  com a query" mesmo nas classes raras, porque ArcFace é state-of-the-art
  em similaridade visual de rostos.
- **Mas ArcFace não separa kin de non-kin globalmente.** Pares de rostos
  visualmente similares mas **não-parentes** ficam misturados com kin no
  espaço de scores, arrastando precision e Test AUC para baixo.
- **Threshold compensação**: o protocolo escolheu threshold 0.20 (baixo)
  para maximizar F1 na validação. Isso eleva recall (incluindo nas classes
  raras) mas destrói precision.

### A síntese

**Kinship verification não é uma única tarefa.** É a interseção de:
- **(a) similaridade facial** — "esses rostos compartilham traços visuais?"
- **(b) ausência de identidade** — "esses rostos NÃO são da mesma pessoa?"

ArcFace é **state-of-the-art em (a)** mas explicitamente treinado **contra
(b)** (afinal seu objetivo é "esses rostos são a mesma pessoa?" — exatamente
o oposto do que kinship requer).

Em regime **frozen**, M08 herda só (a) sem (b). Resultado: alta similaridade
facial recall (mesmo para classes raras como avo-neta, onde diversidade
visual é maior), mas baixa separação global (porque (b) está negada).

Em regime **fine-tuned end-to-end** (como FaCoRNet, M02 R031), o treino
kinship pode **desfazer parcialmente** a pressão anti-kinship do pretreino
ArcFace. As features finais ainda começam de ArcFace, mas se movem para
preservar (b). Por isso M02 R031 funciona (AUC 0.850).

Em regime **frozen + retrieval** (M08), a head não consegue desfazer (b) no
backbone — só compensa parcialmente via cross-attention sobre supports. O
sinal de identidade-anti-família já está cristalizado no backbone.

### Implicação para o projeto

| Encoder regime | Sinal (a) sim facial | Sinal (b) anti-identidade | AUC esperado |
|---|---|---|---:|
| ImageNet ViT frozen | Médio | Neutro (sem treino face) | 0.78 (M06 R001) |
| DINOv2 frozen+LoRA | Médio | Neutro | 0.81 (M05 R001) |
| **ArcFace frozen** | **Alto** | **NEGATIVO** | **0.69 (M08 R001) ←** |
| ImageNet ViT fine-tuned | Médio→Alto | Neutro→positivo | **0.85 (M02 R031)** |
| ArcFace fine-tuned (FaCoRNet) | Alto | Pode aprender positivo | ~0.85-0.87 (literatura) |

O regime **frozen + face-specific encoder** é o **pior** dos quadrantes.
O regime **fine-tuned + face-specific** é o melhor, mas isso é literalmente
FaCoRNet — não há contribuição original em re-implementar.

### Conclusão para o TCC

> "M08 R001 produz Test ROC-AUC = 0.693, o pior resultado do projeto, e
> **−0.083** abaixo do mesmo arranjo arquitetônico com encoder visual
> genérico (M06 R001). Três variantes de ArcFace (IResNet-50 CASIA-FaceV5,
> IResNet-100 Glint360K, IResNet-100 MS1MV3) reproduzem o mesmo padrão de
> degradação, confirmando que o resultado não depende da qualidade
> específica do pretreino face. Atribuímos a falha ao objetivo de
> discriminação de identidade no pré-treino do ArcFace, que ativamente
> separa identidades distintas — incluindo membros da mesma família —
> empurrando familiares para regiões distantes do espaço de embedding em
> regime frozen.
>
> Surpreendentemente, M08 R001 exibe **recall significativamente melhor que
> M05 R001 nas classes raras de avó/avô** (gmgd 71.5% vs 40.7%, gfgs 71.4%
> vs 39.8%) — sugerindo que features ArcFace **retêm** sinal facial útil
> para retrieval mesmo de parentes distantes, mas o **mecanismo de
> threshold global** falha em separar kin de non-kin similares. Essa
> dissociação entre recall per-classe e AUC global indica que **kinship
> verification não é uma única tarefa**: é a interseção de (a)
> similaridade facial e (b) ausência de identidade, e ArcFace está bem
> treinado para (a) mas explicitamente contra (b)."
